# Task Sizing And Dependencies

## Task 粒度

### 太大

- 超過 2 天仍是單一 task
- 同時混多種責任
- Done criteria 模糊

### 合理

- 2-8 小時可完成
- 單一主要輸出
- 一個 owner 可負責
- 可驗證、可追蹤

### 太小

- 小於 1 小時但被拆成大量碎片
- 管理成本高於任務本身

## Dependency 原則

- 先寫清楚 blocked by / enables what
- 能並行的不要假裝串行
- 需要先穩定介面或 schema 的工作放前面
- 測試、QA、Security Review 不是隱形工作，要進計畫
