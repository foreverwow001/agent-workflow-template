# Refactor Workflow

## 安全流程

1. 定義要保留的行為與不可破壞的輸出。
2. 找出當前最主要的 code smell，不要同時清十種問題。
3. 選擇最小、最可逆的一步開始。
4. 做完一步就驗證。
5. 若任何一步引入行為變更風險，先停下來重估。

## 什麼時候不該繼續 Refactor

- 沒有任何驗證手段
- 你已經開始改動對外契約，但 Spec 沒有授權
- deadline 極緊且目標是熱修，不是可維護性改善
- 問題根本不是結構，而是需求不清或邏輯錯誤

## 最後自檢

- 行為是否保持不變？
- 修改是否可以被 QA 解釋與驗證？
- 是否移除了至少一個明確的維護痛點？
- 是否避免順手加入新功能？
